# CalenZoey PRD

## 1. 产品定位

CalenZoey 是一个面向“容易焦虑但希望稳定照顾自己”的个人生活规划工具。它以周历和日历为核心，帮助用户在工作、运动、知识汲取、inner peace、社交活动、饮食和健康数据之间建立一个温柔但可执行的生活节奏。

产品不是为了制造更多待办压力，而是为了让用户每天打开时能快速知道：

- 今天要做什么。
- 做到什么程度就已经很好。
- 饮食和运动是否大致支持减脂目标。
- 本周/月哪些事情真的发生了。
- 下周如何更平衡地安排，而不是靠临场焦虑做决定。

## 2. 用户画像

核心用户为单人使用：

- 工作时间固定：工作日 08:30-11:30、14:00-17:00。
- 早上通常不额外安排事情。
- 当前目标包含减脂、运动规律、知识补齐、身心平衡、社交和生活体验。
- 每天佩戴 Apple Watch，希望利用真实运动和消耗数据修正计划。
- 愿意提供 LLM API，例如 Kimi，用于规划建议、饮食估算、复盘和个性化月报。
- 需要电脑和手机都能用，先接受本机服务器方案，未来希望接近 App 体验。

## 3. 核心目标

### 3.1 MVP 目标

- 支持周日规划下一周，从周一到周日。
- 支持日历式查看一周和一天的活动。
- 支持每天结束时记录完成情况。
- 支持活动分类、子类型、计划时间、实际完成、主观感受。
- 支持记录饮食，包含文字、照片入口和估算热量。
- 支持每日热量差目标，默认制造约 300 kcal 缺口。
- 支持月度总结，按项目统计完成量，并生成个性化可爱月报。

### 3.2 后续目标

- 接入 Apple Health/Apple Watch 同步实际运动数据。
- 接入 Kimi 多模态模型，根据饮食照片估算食物和热量。
- 接入 LLM 进行周计划建议、日复盘、月报生成。
- 支持 PWA 或移动端壳应用，让手机端接近原生 App 使用体验。
- 长期支持更智能的推荐，例如根据疲劳、睡眠、运动负荷动态调整计划。

## 4. 非目标

- 不做多人协作。
- 不做严肃医疗、诊断、处方或疾病管理。
- 不追求卡路里估算的医学级精度。
- 不在 MVP 阶段直接发布到 App Store。
- 不在前期复杂接入第三方商业饮食数据库，先以用户输入和 LLM 估算为主。

## 5. 信息架构

### 5.1 主导航

- 今日：今天的时间线、饮食记录、运动同步、晚间复盘。
- 周计划：周一到周日的 calendar 视图，支持周日规划。
- 饮食：每日餐食、照片、估算热量、备注。
- 运动与健康：计划运动、实际运动、消耗、Apple Health 同步状态。
- 月报：月度统计、项目完成量、可爱月报。
- 设置：个人偏好、目标、API Key、HealthKit 同步设置、数据导入导出。

### 5.2 Calendar 视图

默认展示一周：

- 列：周一到周日。
- 行：中午、下午后、晚上、全天/灵活事项。
- 工作时间块默认固定显示但不制造待办压力。
- 活动以颜色和图标区分类别。
- 每个活动展示计划状态：计划中、完成、部分完成、跳过、自动导入。
- 支持拖拽调整时间，支持复制、延后、标记完成。

## 6. 活动分类

### 6.1 上班

- 固定时间：08:30-11:30、14:00-17:00。
- 不作为压力型待办，主要用于规划避让和能量占用。

### 6.2 运动

子类型：

- 无氧：臀腿、肩背。
- 舞蹈：hiphop、jazz、urban、locking、popping 等。
- 游泳。
- 攀岩。
- 有氧。

规划规则：

- 每天安排 1-2 次运动。
- 工作日五天中，尽量有两天安排 2 次运动。
- 周一不安排 2 次运动，因为周一倒时差。
- 如果中午已安排运动，晚上可以不安排。
- 两次运动通常为有氧 + 无氧。
- 游泳可以和其他有氧结合，但需要避免过度疲劳。
- 周六晚上选择下一周舞蹈课，周日做完整周计划时使用该信息。

### 6.3 知识汲取

子类型：

- 前沿 LLM 进展。
- 开发知识。
- 金融行业/美股。
- 照片处理。
- 产品思维与 vibe coding idea。

设计原则：

- 以小块、低阻力为主，例如 25-45 分钟。
- 支持输入“今天只想看 1 篇文章/1 个 demo/1 个概念”。
- 支持记录收获，而不是只记录时长。

### 6.4 Inner Peace

子类型：

- 深度小说，偏东亚尤其中文小说。
- 工具类书籍，例如亲密关系、心理韧性、人际沟通、自我接纳。
- 电影。
- 电视剧。

设计原则：

- 不把休息做成 KPI。
- 记录“被安抚/被触动/被启发”的感受。
- 月报里突出“本月让我平静下来的内容”。

### 6.5 社交活动

子类型：

- 吃饭。
- 北京市内短行。
- 宿外长途。

设计原则：

- 支持提前规划时间、地点、同行人、预算、交通。
- 支持复盘消耗和获得感，避免只从“是否耽误计划”评价社交。

### 6.6 饮食与减脂

功能：

- 每天记录早餐、午餐、晚餐、加餐。
- 支持文字输入、照片上传、备注。
- 支持 LLM 根据照片和文字估算食物种类、份量、热量和置信度。
- 支持用户手动修正估算结果。
- 每日目标为约 300 kcal 热量缺口。
- 支持展示：摄入热量、运动消耗、基础/日常消耗估计、当前差值。

注意：

- 照片估算卡路里可行，但只能作为近似值。产品必须把它设计成“估算 + 用户确认”，不能当作绝对事实。

## 7. 核心流程

### 7.1 周六晚舞蹈课选择

1. 用户进入“舞蹈课候选”。
2. 记录下周可选课程：时间、舞种、强度、地点、老师、偏好。
3. 选定课程后进入“待纳入周计划”。
4. 周日规划时，系统优先锁定这些课程。

### 7.2 周日计划下一周

1. 系统展示固定工作时间和已锁定事项。
2. 用户选择本周重点，例如减脂、恢复、开发补课、社交。
3. 系统根据规则生成初稿：
   - 周一轻量。
   - 工作日两天双运动，但避开周一。
   - 中午优先安排运动或短知识块。
   - 晚上保留 inner peace 和弹性空间。
4. 用户拖拽调整。
5. 系统检查冲突、运动负荷、热量目标和恢复空间。
6. 用户确认后生成本周计划。

### 7.3 每日执行

1. 今日页展示今天的时间线。
2. 到达计划时间前可轻提醒。
3. 用户完成后标记完成/部分完成/跳过。
4. 饮食随时拍照或文字记录。
5. 晚上进行 2-5 分钟复盘：
   - 今天完成了什么。
   - 哪件事让我感觉更好。
   - 饮食是否大致 OK。
   - 明天是否需要微调。

### 7.4 月度总结

1. 系统按分类汇总数量、时长、完成率、连续性。
2. 展示运动总时长、无氧次数、舞蹈次数、游泳次数、有氧次数、攀岩次数。
3. 展示知识汲取主题分布和产出。
4. 展示 inner peace 内容清单和情绪标签。
5. 展示社交和出游记录。
6. 展示饮食记录完整度、平均热量差、达成 300 kcal 缺口的天数。
7. LLM 生成个性化月报，语气可爱、温柔、不过度鸡血。

## 8. LLM 能力设计

### 8.1 周计划助手

输入：

- 固定工作时间。
- 上周完成情况。
- 本周目标。
- 已锁定事项。
- 运动规则。
- 疲劳/时差/经期/天气等可选状态。

输出：

- 一周计划草案。
- 每个安排的理由。
- 可替代方案。
- 风险提醒，例如运动过密、恢复不足。

要求：

- LLM 只生成建议，不直接写入最终计划。
- 服务端必须校验时间冲突和规则约束。

### 8.2 饮食照片估算

输入：

- 餐食照片。
- 用户补充文字，例如“半碗米饭”“鸡腿去皮”“奶茶无糖”。

输出：

- 食物列表。
- 估算份量。
- 估算热量。
- 蛋白质/碳水/脂肪粗略估算。
- 置信度。
- 需要用户确认的问题。

要求：

- 明确展示“估算值”。
- 允许用户快速修正份量和热量。
- 修正值优先于模型估算进入统计。

### 8.3 日复盘助手

输入：

- 今日计划与完成。
- 饮食记录。
- 运动记录。
- 用户主观情绪。

输出：

- 温柔复盘。
- 明天 1-3 个微调建议。
- 不使用羞辱、惩罚、内疚驱动语言。

### 8.4 月报助手

输入：

- 月度聚合数据。
- 关键事件。
- 用户偏好的表达风格。

输出：

- 可爱月报标题。
- 本月亮点。
- 分类统计。
- “照顾到了自己的证据”。
- 下月建议。

## 9. 数据指标

### 9.1 规划指标

- 每周计划创建次数。
- 计划活动数量。
- 计划变更次数。
- 完成/部分完成/跳过数量。

### 9.2 运动指标

- 总运动时长。
- 运动类型分布。
- 双运动日数量。
- 周一是否避免双运动。
- Apple Health 导入运动与手动计划匹配率。

### 9.3 饮食指标

- 饮食记录天数。
- 照片记录数量。
- 平均摄入估算。
- 平均热量缺口。
- 达到 300 kcal 缺口天数。

### 9.4 心理友好指标

- 晚间复盘完成天数。
- 用户主观情绪趋势。
- “今天足够好”标记次数。
- 计划过载提醒次数。

## 10. 体验原则

- Calendar 是主界面，但不要像工作软件一样压迫。
- 所有目标都允许“部分完成”。
- 对跳过事项给出中性记录，不使用失败语言。
- 复盘文案温柔、具体、行动导向。
- 统计展示成“证据”，不是审判。
- 默认每周至少保留一些空白时间。
- 手机端以快速记录为主，电脑端以规划和复盘为主。

## 11. MVP 验收标准

- 用户可以在本机启动网页并访问。
- 可以创建、编辑、删除一周计划。
- 可以按类别和子类型记录活动。
- 可以完成每日复盘。
- 可以上传或选择饮食照片，录入文字说明。
- 可以手动录入热量，并计算每日热量差。
- 可以生成月度分类统计。
- 可以调用 Kimi API 生成周计划草案和月报文本。
- 所有核心数据本地持久化。
